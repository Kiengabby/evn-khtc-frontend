# API Response vs UI Display - Data Validation Analysis

## 📊 Issue Summary
You're seeing only **STT numbers 1 and 2** in the grid, but the API response contains **6 data rows** (R3-R8).

---

## 🔍 Root Cause Analysis

### 1. **API Response Data Issue** ❌
The API returns cell data with ALL STT cells having `value: 0`:

```json
{
    "rowCode": "SL_GIA_THANH",
    "colCode": "STT",
    "value": 0,              // ← Wrong! Should have meaningful value
    "isReadOnly": true
},
{
    "rowCode": "CP_THEO_YT",
    "colCode": "STT",
    "value": 0,              // ← Wrong!
    "isReadOnly": true
},
```

**Expected**: STT column cells should contain values like `1`, `2`, `3`, etc., or be empty for child rows.

---

### 2. **Row Hierarchy in API Response** 📋
The API defines rows with different levels:

```json
"rows": [
  {"rowKey":"R3", "rowCode":"SL_GIA_THANH", "title":"Sản lượng tính giá thành", "level":0, "isReadOnly":false},
```

### 3. **Frontend Rendering Logic** 
File: `layout-grid-renderer.service.ts` → Line 165-178

```typescript
private buildDataRows(visibleCols, bodyRows) {
  let sttCounter = 0;
  
  for (const rowDef of bodyRows) {
    // STT is ONLY assigned to level === 0 rows
    if (sttIdx >= 0) {
      if (rowDef.level === 0) {  // ← Only parent rows!
        sttCounter++;
        row[sttIdx] = rowDef.isReadOnly ? '' : String(sttCounter);
      }
    }
    // ...
  }
}
```

**Logic**: Only rows with `level === 0` get STT numbers. This is why you see **1 and 2** only.

---

## ✅ **Current Behavior is CORRECT** (by design)

The system is correctly implementing:
- **Hierarchical grid**: Parent rows (level 0) get sequential numbers
- **Child rows** (level 1, 2): No STT number, indented display
- **API issue**: All STT cells have `value: 0` (should be empty or meaningful), but this is **overridden** by the frontend logic anyway

---

## 🛠️ **What Needs Fixing**

### **Issue #1: Backend - API returns wrong STT values** 🔴

**Location**: `/api/v2/PlanningData/load-form`

**Problem**: 
```json
{
  "rowCode": "SL_GIA_THANH",
  "colCode": "STT",
  "value": 0  // ← All cells have 0
}
```

**Should be**:
- For parent rows (level 0): Empty string or omit from response
- For child rows (level 1, 2): Always empty (frontend will handle numbering)

OR better: **Don't include STT and CHITIEU_NAME cells in the API response at all** — these are generated by the frontend based on the row structure.

**Fix**: Update backend to either:
1. Return empty values for label columns (STT, CHITIEU_NAME, UNIT/DVT)
2. Or omit them entirely from the cells array

---

### **Issue #2: Backend - Label columns should not be in cells array** 🔴

The API includes cells for columns that should only be generated by frontend:
- `STT` (sequential number)
- `CHITIEU_NAME` (row title with indent)
- `UNIT` / `DVT` (unit of measurement)

**Current API response includes**:
```json
{
  "rowCode": "SL_GIA_THANH",
  "colCode": "STT",
  "value": 0,
  "isReadOnly": true
},
{
  "rowCode": "SL_GIA_THANH",
  "colCode": "CHITIEU_NAME",
  "value": 0,
  "isReadOnly": true
},
```

**Better approach**: Only send actual data cells, let frontend generate labels.

---

## 📝 **Data Consistency Recommendations**

### **For Backend API** 
1. ✅ Do NOT include label column cells (STT, CHITIEU_NAME) in the response
2. ✅ Set STT and row title cells to empty `value` if you must include them
3. ✅ Return only data cells that need to be edited/saved

### **For Frontend** 
Current implementation is correct:
- ✅ Generates STT numbers automatically
- ✅ Formats row titles with indentation
- ✅ Ignores API values for label columns
- ✅ Populates only data cells from API
